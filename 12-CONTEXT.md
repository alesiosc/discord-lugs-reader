+# Verify Before Declaring Done
+
+> Loaded automatically via SOUL.md rule #7 on every build task. Must be followed step-by-step.
+
+## When to Load
+
+After every feature build, fix, integration, or refactor — before telling the user it's done.
+
+## Verification Checklist
+
+### 1. Wiring Audit
+- [ ] Every new file is **imported** somewhere — no orphan modules
+- [ ] Every new class/function is **registered** (routes, handlers, event listeners, DI container, etc.)
+- [ ] All config entries / env vars referenced in code exist in the actual config files or `.env`
+- [ ] Startup hooks / initialization sequences include the new component
+- [ ] Event bus subscriptions are actually wired, not just defined
+
+### 2. Trace the Execution Path
+- [ ] From request entry point → through middleware/handlers → to response — every hop is connected
+- [ ] No dangling `TODO` or `# placeholder` that skips a real implementation step
+- [ ] No "stub" functions left in production paths
+
+### 3. End-to-End Test
+- [ ] Run the actual feature (not unit tests alone — real execution)
+- [ ] Verify output matches what the user asked for
+- [ ] Check error paths: what happens when a dependency fails or returns nothing?
+
+### 4. Configuration & Environment
+- [ ] All required env vars are documented or set
+- [ ] Config files reference the new components by their correct names (no typos in module names)
+- [ ] Paths used in config match actual filesystem paths
+
+### 5. Read the Original Request
+- [ ] Re-read the user's original request
+- [ ] Does the output actually match what was asked? Every requirement covered?
+- [ ] Nothing added that wasn't asked for? Nothing missing?
+
+### 6. Final Sanity
+- [ ] Would this work if the user ran it fresh, with no knowledge of what I did?
+- [ ] Have I left any manual steps the user needs to run? If so, document them clearly.
+- [ ] Is there any implicit assumption that "the user will figure out the rest"?
+
+## If Anything Fails
+
+**DO NOT** declare done. Fix it. If you can't fix it, report what specifically is still broken and why — don't gloss over it.
+
+## Pitfalls
+
+- "It compiles" / "no syntax errors" is NOT verification. Runtime wiring failures don't raise syntax errors.
+- Unit tests passing in isolation does NOT mean integration works.
+- If something is "obvious" or "trivial" to wire up, that's exactly the thing most likely to be forgotten. Check it explicitly.
+- The user has explicitly stated this is the #1 recurring failure mode. Treat this checklist as binding.
